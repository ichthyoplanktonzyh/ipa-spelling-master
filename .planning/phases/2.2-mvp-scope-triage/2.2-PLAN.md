# Phase 2.2: MVP Scope Triage

## 目标

在继续实现结果页和训练历史前，基于产品目标和 DDD 边界，明确哪些功能属于 MVP 必要闭环、哪些应延后、哪些是伪需求，形成可执行的范围约束。

## 任务

- [ ] 2.2.1 梳理需求清单：从 `REQUIREMENTS.md`、`PROJECT.md`、`ROADMAP.md`、`DDD-ARCHITECTURE.md` 中抽取当前 M2/M3/M4 相关功能
- [ ] 2.2.2 建立判定标准：以“选择 L2 → 完成训练 → 获得反馈 → 查看结果 → 再练/换主题”为核心闭环
- [ ] 2.2.3 分类功能：标记为 MVP 必要、必要但不急、远期增强、伪需求/明确不做
- [ ] 2.2.4 更新 `REQUIREMENTS.md`：为需求增加 MVP scope 标记或说明，不修改稳定 ID
- [ ] 2.2.5 更新 `ROADMAP.md`：确保 M2/M3/M4 排期与 scope triage 结果一致
- [ ] 2.2.6 输出后续实现约束：为 Phase 2.3 Feedback & Session Results 明确“做什么/不做什么”

## 退出条件

- [ ] 每个 M2 相关需求都有明确 scope 分类
- [ ] 明确列出本阶段不做的伪需求，并说明原因
- [ ] Phase 2.3 的输入范围清晰，不再混入账号、后端、ASR、社交、课程化等非 MVP 内容
- [ ] `REQUIREMENTS.md`、`ROADMAP.md`、`STATE.md` 对下一步工作描述一致

## 影响文件

- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `.planning/STATE.md`
- `.planning/codebase/DDD-ARCHITECTURE.md`
- `.planning/phases/2.3-feedback-session-results/2.3-PLAN.md`

## 约束

- 不改需求 ID，只追加分类/说明
- 不做代码实现
- 不把远期增强删除，只调整到正确里程碑
- 伪需求必须有判定理由，避免以后反复进入 scope
